# Step 3: Roadmap

With your design documents in place, Step 3 is about making decisions: what to build first, what to build later, and what not to build at all (yet).

A roadmap isn't a guarantee — it's a plan that evolves. The goal is to sequence your work intentionally rather than reactively.

---

## The Three-Tier Model

| Tier | What it means |
|------|--------------|
| **MVP** (Minimum Viable Product) | The smallest version of the product that proves the core value proposition — nothing extra |
| **V1** | The first full public release — MVP plus the features needed for it to be genuinely useful |
| **V2+** | Enhancements, scaling, and "nice to haves" — built after you've validated V1 with real users |

The most common mistake is making the MVP too big. If the MVP requires months of work, it's not minimal. Push everything that isn't strictly necessary to V1 or V2.

---

## How to Work Through This Step

1. Read all Step 2 design documents (especially `MARKET_FIT.md` and `USER_JOURNEYS.md`)
2. List every feature you want to build
3. For each feature, ask: "Can the product prove its value without this?" If yes, it's not MVP.
4. Assign every feature to a tier
5. For MVP and V1, define milestones and rough timelines

---

## Done Enough to Move On?

You're ready for Step 4 when:
- Every planned feature is assigned to a release tier (MVP, V1, V2+)
- The MVP scope is small enough to actually build and test
- Milestones for the current tier have been defined

---

## Claude Guidance for This Step

When helping a user at Step 3:
- Read all Step 2 documents before starting — the roadmap should be grounded in design decisions already made
- Push back on MVP scope. "What's the minimum that proves this works?" is the key question.
- When a user wants to include something in the MVP "just in case", help them evaluate: does this add risk, or does it validate the core premise?
- For each V2+ item, capture the reasoning briefly — not as a promise, but so the user remembers why they deferred it
- Milestones should be outcome-based ("users can complete a purchase") not task-based ("implement checkout UI")
