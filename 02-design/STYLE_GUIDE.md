# STYLE_GUIDE.md

**Level 2 — Design** | Contributed by: Product owner, design collaborators

This document defines the visual and verbal identity of your product — colors, typography, tone, component patterns, and brand values. It keeps the product feeling consistent across screens, sessions, and contributors.

You don't need a professional design background to fill this out. You need opinions about how your product should feel.

---

> **Claude Guidance:** Start by asking the user: "What 3 words would you want someone to use to describe how this product feels?" and "Are there products or websites whose look and feel you admire?" Use their answers to infer direction before getting into specifics. Keep this document practical — focus on decisions that Claude will actually reference when generating UI code or copy. Avoid prescribing exact hex codes or pixel values unless the user already has them; use descriptive language ("warm neutral background", "bold sans-serif headline") that can be refined later.

---

## Brand Voice and Tone

*How should this product sound? Describe the personality — formal or casual, playful or serious, technical or accessible. Give an example of a sentence that fits the tone and one that doesn't.*

## Visual Identity

### Color Palette

*Primary, secondary, and accent colors. If you don't have hex codes yet, describe the feeling: "deep blue for trust, warm orange for calls to action."*

### Typography

*Headline and body font choices, or the style of typeface (e.g., "clean modern sans-serif", "humanist with warmth"). Note if there are size or weight conventions.*

### Spacing and Layout

*How dense or spacious should the UI feel? Any patterns for margins, padding, or grid structure.*

### Iconography and Imagery

*Should the product use icons? What style (outlined, filled, illustrated)? Are there image guidelines (photography style, illustration)?*

## Component Patterns

*Conventions for common UI elements: buttons (when to use primary vs. secondary), forms (label placement, error display), navigation (where menus live), modals (when to use them).*

## Accessibility Standards

*Minimum contrast ratios, keyboard navigation expectations, screen reader considerations, and any specific accessibility commitments.*
