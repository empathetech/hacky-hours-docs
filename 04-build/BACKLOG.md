# BACKLOG.md

**Step 4 — Build** | Maintained by: Product owner, updated continuously

The backlog is the ordered list of tasks that need to be completed to reach each release. Tasks are discrete, implementable units of work — not vague goals. Each task maps to a branch name, a semantic version, and a design document that justifies it.

---

> **Claude Guidance:** When helping the user populate this backlog, translate roadmap features into concrete, implementable tasks. A good task: has a clear definition of done, maps to a specific design document, and can be completed in one focused session. If a "task" requires multiple sessions to complete, break it down. Tasks should be ordered within each version — dependencies first. Before starting any task, read the linked design document section. After completing a task, help the user write the commit message and CHANGELOG entry.

---

## How to Read This Backlog

- **Status**: `[ ]` = pending, `[~]` = in progress, `[x]` = complete
- **Branch**: the git branch name for this task
- **Version**: the semantic version this task ships in
- **Design ref**: the document section that defines the requirement
